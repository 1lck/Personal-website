2025 年暑期，我在武汉赢派有限公司技术研发部担任 Java 全栈实习生。入职的第一天就接到“拆弹”任务：为 B 端旅游系统“趣游兔”重构多租户架构。系统采用 Spring Boot 3.4.6 + Vue3，八大业务域共 180 余张表，只要有一张漏写租户条件就会发生数据串租。以下是我分阶段的实战笔记。

## 目标拆解

我把目标分成三类：**逻辑隔离、性能保障、运维可观测**。对应指标是：

1. 核心表租户字段覆盖率 100%，API 层自动注入，禁止人工遗漏；
2. Redis、权限、搜索等周边系统与租户保持一致；
3. 故障时 5 分钟内定位问题租户、请求链路和操作者。

## 数据隔离：字段 + 拦截器

- 在订单、行程、库存等 46 张核心表新增 `tenant_id` 字段，并将历史数据回填。
- 编写 MyBatis 拦截器，对 Insert/Update/Delete/Select 统一注入租户条件；若开发者绕过 Mapper 会直接抛异常。
- Sa-Token 自定义 `TenantContext`，把租户信息从网关一路透传到 DAO 层，任何线程切换都会重新绑定。

## 缓存与权限

- 原有 Redis 只有全局 key，缓存污染严重。我设计 `tenant:{id}:业务Key` 的命名规范，配合 Lua 脚本保证批量写入原子性。
- 权限体系基于 Sa-Token 的 `StpInterface` 扩展，将菜单、按钮、数据级权限拆分为“系统默认 + 租户增量”两层。
- 引入租户级配置中心，允许某些租户开启自定义流程，让产品上线后无需切代码。

## 灰度、监控与压测

- 使用 SkyWalking + Logback MDC 记录租户 ID、TraceId、用户 ID，构成排障三件套。
- Gatling 脚本模拟 2 万并发写入，验证在多租户场景下 TPS 和时延；并用 ChaosBlade 进行 Redis 故障演练。
- 上线前在生产环境“灰度租户”跑了一周，期间发现 3 个 SQL 遗漏字段，通过自动校验脚本及时修复。

## 结果与复盘

正式发布后，系统再没出现串租事故，缓存命中率提升到 87%，平均响应时间稳定在 120ms 以内。最重要的收获是：架构工作并不只写方案，而是要贯穿需求澄清、编码、联调、灰度和监控的全链路。我后来在学校的竞赛管理系统里也复用了这套方法，效果同样可靠。希望这篇复盘能给同样在做多租户改造的同学一些参考。
