“趣游兔”最吸引用户的功能是 VR 全景看景点，但早期架构里，资源加载慢、热点编辑困难、后端接口也缺乏缓存。接手后我用两周时间对服务端和前端架构做了一轮“翻新”，核心还是依靠 Spring Boot + Redis + Krpano 的组合。

## 静态资源管线：先瘦身再分层

- 旅游运营同学上传的全景原图动辄 12MB，我通过 Spring Boot 的定时任务调用 Thumbnailator 批量压缩，并生成 4 级分辨率切片。
- 切片信息写入 PostgreSQL，同时在 OSS 上按 `scene/{tenant}/{resolution}` 分类，方便 CDN 缓存。
- 访问时控制器优先返回低分辨率地址，前端拿到 token 后再分批拉取高清图，首屏平均耗时从 3.2s 降到 1.8s。

## Krpano 渐进式加载

- 后端提供 JSON 配置动态控制 Krpano 的 `progressive=1` 及热点预加载策略。
- 对热点资源采用“视角+半径”命中策略，用户只有在接近目标点时才获取详细贴图。
- Redis 记录热点热度，配合定时任务刷新 CDN，用户高峰期访问也不会挤爆带宽。

## 热点编辑体验

- 旧版只能手输坐标，运营老师直接崩溃。我在 Vue3 端实现热点拖拽，后端提供坐标校验与批量导入/导出接口。
- Spring Boot 通过 WebFlux 支持大文件导入上传，操作完成后写入 MQ 通知审核人。

## 监控与回退

- 利用 Micrometer + Prometheus 采集每个场景的加载时长、失败率，Grafana 实时展示。
- 当某个场景资源损坏时，服务端会降级返回低清晰度的备用图，并通过企业微信提醒编辑人员。

最终，VR 模块的平均加载时长下降 35%，热点配置效率提升 70%，业务同事不用再频繁找开发帮忙。更重要的是，我把这套“Java 后端 + 3D 前端”的协同流程吃透了，后续在校园可视化项目中也直接复用这套模式。
